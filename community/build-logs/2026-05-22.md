# Bizbox Build Log — Week of 2026-05-22

Nine releases shipped this week (v0.0.21 → v2026.522.0), 11 substantive PRs merged (10 merged within the window; one merged the prior week but released on May 18). The theme: making the control plane legible — to operators watching their agent pipelines, and to the agents themselves reconciling approval state.

---

## Shipped This Week

### 🗺️ Issue Graph View — see your agent pipeline as a graph
**PR [#58](https://github.com/zesthq/bizbox/pull/58) · v0.0.23 · @DennisDenuto**

The inbox and issue detail views have always shown parent/child relationships, but navigating a multi-agent execution flow still meant clicking through individual issue pages. The new graph view renders the full issue hierarchy — parent, children, related work, agent assignments, and produced deliverables — as a visual pipeline. Operators can now see at a glance where work is, who touched it, and what came out of it.

This is the first step toward making Bizbox's execution model inspectable without reading raw issue lists.

### 🏷️ Multi-entity mention tagging in the editor
**PR [#61](https://github.com/zesthq/bizbox/pull/61) · v0.0.22 · @ralphbibera**

`@` mentions in issue comments and descriptions now support scoped tagging for Issues, Deliverables, and Projects — not just agents. The markdown editor renders these as typed mention chips, and the backend resolves them to first-class entity links. Cross-entity referencing is now consistent and fast across all writing surfaces.

### 📥 Inbox: related work nested under parent rows
**PR [#57](https://github.com/zesthq/bizbox/pull/57) · v0.0.22 · @DennisDenuto**

Parent issues in the inbox now expand to show not just child issues but also their inbound and outbound references as second-level nested rows. Before this change, related work was only visible in the issue detail view, making the inbox an incomplete picture for top-level items with important cross-references. No changes to existing inbox filtering or top-level visibility rules.

### 🔔 Awaiting-human notification outbox (merged prior week, released May 18 in v0.0.21)
**PR [#56](https://github.com/zesthq/bizbox/pull/56) · v0.0.21 · @adPalafox**

Human-approval notifications to ClickUp now go through a durable outbox table before delivery. The outbox handles deduplication, retries, and attaching review files and ClickUp task links. This replaces a fire-and-forget approach that could silently drop notifications when the ClickUp API was slow or the agent run ended before delivery confirmed.

*Note: PR #56 merged on May 15 (prior week). It shipped to production in v0.0.21, released May 18 — the first release of this week's window.*

### ⚙️ ClickUp approval channel configuration
**PR [#65](https://github.com/zesthq/bizbox/pull/65) · v2026.521.2 · @adPalafox**

Two new env vars — `CLICKUP_AWAITING_HUMAN_CHANNEL_ID` and `CLICKUP_AWAITING_HUMAN_CHANNEL_NAME` — let operators configure exactly which ClickUp channel receives awaiting-human approval messages. The previous behavior (falling back to a hardcoded channel) is preserved for backward compatibility. Message content was also improved to include more context for the human reviewer.

### 🩹 Heartbeat: approval message ID race condition fixed
**PR [#68](https://github.com/zesthq/bizbox/pull/68) · v2026.522.0 · @adPalafox**

When the heartbeat service reconciles a pending awaiting-human approval, it needs the ClickUp message ID to match the right notification. There was a race: if the activity log was written before the outbox delivered the notification, the heartbeat couldn't find the message ID and would stall. The fix adds a fallback that resolves the message ID directly from the outbox table, closing the race.

### 🔁 Heartbeat: terminal "ok" payloads treated as authoritative success
**PR [#66](https://github.com/zesthq/bizbox/pull/66) · v2026.521.3 · @adPalafox**

The heartbeat recovery logic now recognises terminal "ok" payloads as definitive success even when the adapter returns a non-zero exit code or an error message alongside the payload. Previously, a non-zero exit code could trigger unnecessary recovery retries on runs that had actually completed successfully. This reduces noise in the recovery queue and avoids double-execution risk.

### 🔑 OpenClaw adapter: persist pairing keys on agent updates
**PR [#59](https://github.com/zesthq/bizbox/pull/59) · v0.0.21 · @ralphbibera**

The saved-agent update route was not persisting `devicePrivateKeyPem`, which meant an existing agent switched into device-pairing mode would generate a new ephemeral identity on every run. The create and join-default flows already handled this correctly; the update path now does too.

### 📄 Smarter filenames for document deliverables (two-part improvement)
**PRs [#62](https://github.com/zesthq/bizbox/pull/62) + [#64](https://github.com/zesthq/bizbox/pull/64) · v0.0.24 + v2026.521.0 · @adPalafox**

Two back-to-back PRs tightened up how Bizbox names document deliverables. PR #62 (v0.0.24, May 20) made filenames prefer a slugified title over a generic key — so instead of `doc-abc123.md` you get `build-log-draft-week-of-2026-05-22.md`. PR #64 (v2026.521.0, May 21) extended that by inferring the title from the first Markdown H1 when no explicit title is stored, so even untitled documents produce meaningful filenames. Non-Markdown documents and Markdown without an H1 still fall back to the generic key.

## Decisions

**Versioning scheme changed mid-week.** Releases shifted from `v0.0.x` to a date-based scheme (`v2026.521.x`) starting with v2026.521.0 on May 21. The new scheme encodes the release date directly in the tag, making it easier to correlate a release with the PR that triggered it. No API or schema changes accompanied the version bump.

**Outbox-first for human notifications.** The team chose to route all awaiting-human ClickUp notifications through a persistent outbox rather than delivering inline during the heartbeat run. The trade-off: slightly more infrastructure (a new DB table, a processing loop) in exchange for reliable, auditable delivery. Given that missed approval notifications block agent work, the reliability gain was worth the complexity.

**Rebrand sweep committed.** The decision to sweep remaining Paperclip references from UI copy, docs, tests, and Storybook fixtures was made and executed in a single PR ([#63](https://github.com/zesthq/bizbox/pull/63)) by a new contributor (@cerkiner). The scope was intentionally limited to visible copy — no runtime behavior changed.

---

## Trade-offs

- **Graph view is read-only at launch.** The issue graph view shipped as a visualisation tool only. Operators can inspect the pipeline but cannot reorder, reassign, or act on issues from the graph. That interaction layer is deferred — the priority was getting the legibility win first.
- **Approval channel config is env-var-only.** `CLICKUP_AWAITING_HUMAN_CHANNEL_ID` and `CLICKUP_AWAITING_HUMAN_CHANNEL_NAME` are set at deploy time, not per-company or per-agent. That's fine for single-tenant deployments but will need revisiting as multi-tenant use cases grow.
- **Filename inference is best-effort.** The two-part filename improvement ([#62](https://github.com/zesthq/bizbox/pull/62) + [#64](https://github.com/zesthq/bizbox/pull/64)) covers the common case well, but H1-based title inference only fires for Markdown documents with a top-level heading. Non-Markdown documents and Markdown without an H1 still fall back to a generic key. Edge cases are documented in the PRs but not yet handled.

---

## Open Challenges

- **Approval reconciliation is still fragile at the edges.** PRs #65, #66, and #68 all touched the awaiting-human approval flow in the same week, which signals the surface is still being hardened. The outbox and message-ID fallback close the known gaps, but the full reconciliation path (outbox → ClickUp delivery → heartbeat poll → interaction accept → agent wake) has a lot of moving parts. More test coverage and an end-to-end integration test are on the backlog.
- **Graph view performance at scale.** The issue graph view fetches related-work enrichment eagerly. For companies with deep issue hierarchies or high issue volume, this could become slow. No pagination or lazy-loading is in place yet.
- **Versioning scheme migration.** The shift to date-based tags mid-week means the changelog and any tooling that parses semver tags will need updating. The build log summarizer already handles both schemes, but downstream consumers (release notes generators, dependency bots) may not.
